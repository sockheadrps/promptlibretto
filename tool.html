Intro — Prompt Libretto & Prompt Constructor

Prompt Libretto is a Python library for building structured, dynamic system prompts — at runtime, from a declarative model called a registry.

The problem it solves is this: as soon as you want a language model's behavior to vary — by persona, mood, context, or conversation state — hardcoding prompt strings stops working. You end up with concatenated strings scattered across your codebase, no consistency, and no way to audit what the model is actually receiving. Prompt Libretto replaces that with a registry: a JSON document that defines every section of a prompt, how sections relate to each other, what's required, what's optional, and what gets selected or randomized at call time. The engine assembles the final prompt from that spec every time.

This tool — Prompt Constructor — is one example of how you can build on top of that. It's a studio for creating chat models: you get a visual builder for authoring registries, a studio for testing them against a live model, and an ensemble mode that runs two registry-driven agents in conversation with each other.

Ensemble is where it gets more interesting. Each participant has memory — an embedding model indexes the conversation as it happens, a summarizer compresses older context into a rolling system summary, and working notes let each agent track its own read on the situation. The scenario is steered — you define the domain, the persona, the tone — but the output is non-deterministic. The model generates freely within those constraints, so each run produces something unique while still staying true to the theme.

Think of it as the difference between scripted dialogue and improv with rules. The registry sets the stage and the character. What actually gets said is up to the model.

Builder Walkthrough — Wizard Tech Support

So this is the Wizard Tech Support registry loaded in Builder.

At the top we have the assembly order — this is the backbone of the system. It defines exactly what sections get included in the final prompt, and in what order. Everything flows from here.

Base context sets the scene — this agent works for a magical helpdesk. There's a template variable for the company name, so you can swap that out per deployment.

Personas give the agent a character. We have two: the Fresh Grad Apprentice — nervous, over-reliant on the script — and the Burned Out Mage, who's done with everyone but still technically doing the job. Each persona has a group of behavioral directives, and the array mode is set to random-one, so it picks a different nudge each time instead of repeating the same behavior.

Sentiment works the same way — two options here, Nervous Helpful and Mystically Reassuring — each with a composure slider that injects a score into the prompt. The groups give it a random behavioral nudge on top of that.

Static injections are facts that are always true — like the supervisor queue being 45 days. You toggle these on or off.

Runtime injections are dynamic — this one has a care level variable you can fill in at call time.

Memory rules are where it gets interesting — if the classifier detects a customer is confused about basic magic, it appends a special ending that changes how the agent responds to that specific message.

And at the bottom, output directions and prompt endings cap it off — format rules and the role label that kicks off the model's response.

Studio Walkthrough — Wizard Tech Support

Now here's the same registry running in Studio — this is where you actually talk to it.

On the left you can see the live state panel. Every section from the assembly order shows up here — persona, sentiment, static injections — and you can change any of them without touching the JSON. Switch personas, move the composure slider, toggle an injection on or off, and the assembled prompt updates instantly.

Down here is the prompt preview. This is exactly what gets sent to the model — no guesswork. You can see the persona context, the behavioral directive that was randomly picked this run, the sentiment nudge, and the static injection text all assembled in order.

I'll type something in as the customer. Notice the model responds in character — polite, slightly panicked, leaning on helpdesk procedure it barely understands.

Now watch what happens when I switch personas to the Burned Out Mage and ask the same kind of question. Same registry, same sentiment slider position — completely different voice. The prompt preview updates and the model responds accordingly.

The output policy is enforced silently in the background — minimum length, forbidden phrases like "As an AI," and automatic retries if the output doesn't pass. You don't see the failures, just the clean result.

And if memory is wired up, working notes and session summaries feed back in here — but we'll get into that separately. The point is Studio lets you audition any combination of settings before you commit anything to the registry file.
